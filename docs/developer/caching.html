Corporate UI caching strategies

When it comes to the caching on an enterprise intranet it differs very much from caching for the commercial web. On an intranet we know for sure that a vast majority are recurring visitors. That's why we can tailor our structure in a way that users not very often need to download all the content of a certain page, rather than just loading the dynamic parts.

Fonts, logos and framework files never expires

We've setup the Corporate UI framework in a way so that static content are separated from dynamic in such a way that it, once downloaded, will remain in the end users local cache memory. This will not only give great benefits but it will also make traditional page speed reports out of order. The page speed reports are in our cases most often not applicable.

All servers delivers the same content

Since most of the really heavy files like the Scania font files, Scania icon set and the Scania logotype package actually are the same ones for all our sites, why then force the user to download it again and again? If all our applications would place for example the Scania Sans symbol on their own servers we would not only create a bad situation for making updates to the logotype possible, we also would force our web browsers to believe that it is a different physical symbol for every time the URL for the symbol changes, even though changing only a sub domain name. Using our dedicated CDN servers, you will share URL with the rest of Scania and that will give you a 95% chance that your users will already have downloaded several megabytes of static content from other servers on Scania. As an example here you'll find the centrally stored wordmark that was already in your cache memory arriving at this page.

https://static.scania.com/resources/logotype/scania/scania-symbol.svg

Preloading at the login page

Since most of our applications demands a log in procedure, we're already at those pages downloading some of our assets. It'll take the user a few seconds to conduct the log in, so at those places we know for sure that we'll have time to preload some generally used files. Therefore, it's important that you'll include your frameworks, libraries, logos, fonts etc. from our server stack at static.scania.com.

static.scania.com

With our CSS, JS, Images and font-files stored in data-centres all over the world, we can ensure the lowest possible response-times for our files. From here you'll get the fastest delivery of the frameworks and libraries that you'll need to inlculed.

Optimization by using HTTP/2

If you seriously need to improve your applications loading time we'd suggest that you'll enable HTTP/2 on your servers. This will make multiplexing possible and that could cut your loading time quite drastically. A demo of the technology can be found at Akamai and a HTTP/2 FAQ is available as well.
