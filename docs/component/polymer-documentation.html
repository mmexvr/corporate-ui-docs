Polymer documentation

Structure

The UX Library is able to dynamically pull and render all Polymer-components under /development/global/html.

The documentation uses <hydrolysis-analyzer> for analyzing the polymer-components. In additions to the analyzer, five elements are used to display the documentation in the UX Library:

  • <scania-doc>
    • <iron-doc-viewer>
      • <iron-doc-property>
      • <demo-component>
      • <marked-element>

The analyzer and the marked-element is unmodified tools from the polymer-catalog. The iron-components are heavily modified versions from the polymer-catalog. <scania-doc> is called in component.html?t=Category&c=Component. The url-parameters is used to generate the URL to the file to be analyzed.

This setup makes some assumptions about the component and has some limitations.

Given a url to a file to analyze, it will analyze any and all Polymer-components it finds in the file - including all HTMLImports. All of these will be displayed in the UX Library. This becomes a potential problem when there are interdependancies! The <hero-carousel>, for example, imports <main-hero> and thus the main-hero will also be analyzed. This behaivour isn't necessarily wanted.

The UX Library and <scania-doc> assumes a certain structure to each component. The UX-Library counts the number of variations of each component by counting the number of files - 1 (accounting for the main component). Files not reletated to a variations should thus be hidden in a subfolder.

<scania-doc> assumes that the folder-name file-name main-component-name are all the same when it sorts the result from the analyzer. If a component-name matches the folder & file-name it will be displayed first. Any components with names inlcuding variation comes next, in ascending order. Any other component shows without sorting at the bottom.

The main-component, or the one showing first, have an API-Reference attached to it. This lists all public properties and methods, including the preceding comment, of the main-component. See the polymer documentation for how to annotate type of params and returns. Private properties/methods should be named with an underscore: _someMethod.

Documentation

Place the documentation directly preceding the <dom-module> as an HTML-comment of each component. Do not place an comment before the Polymer-call in the <script>-tag as this will override the earlier comment-block. Make sure every component has a comment-block, albeit empty. If not, the analyzer will inherit the comment-block of the component preceding it, creating duplicates.

Place all documentation you want hidden from the UX Library after an @dev-tag. Anything written after this tag won't be rendered.

<iron-doc-viewer> uses markdown to render the documentation in the UX Library. If you want to show a demo, put the demo-code inside a <demo></demo>-block. This will render the demo in a full-width container with the code in a accordion-style container following it, should the user want the code.

Cases

Several demos
Seperate the documentation with three underscores. Each created "block" can contain one demo (<iron-doc-viewer> will only render one demo per block).
No variations
Place the main descriptor (one without demos) in the first "block" before any seperator (the three underscores). Place demos after seperator.
